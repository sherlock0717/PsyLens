# 机制编码手册（MECHANISM_CODEBOOK）

> 机制标签用于标注一条反馈**为什么**让玩家产生负面体验（底层心理机制），而非它**在说什么**（表层话题，见 `SURFACE_TOPIC_CODEBOOK.md`）。
> 本手册是 Agent 提案与人工复核的共同依据。标签取值固定为下表六项，不得新增。

## 使用总则

- 一条证据单元优先分配**一个主机制**。
- 当证据文本无法明确指向任一机制、或信息不足时，使用 `uncertain`，不得强行分配。
- `uncertain` 必须真实使用：追求覆盖率而牵强归类会降低编码质量。
- 允许「无机制证据」：纯表层描述、无情绪或诉求指向的短句可判定不构成机制证据（在提案层 `include_as_evidence=no`）。
- 机制判断基于**证据文本本身**，不基于对作者身份或平台的假设。

## 标签总览

| 标签 | 中文名称 | 一句话 |
| --- | --- | --- |
| competence_frustration | 能力挫败 | 玩家觉得「打不过 / 做不到 / 投入无回报」，胜任感受挫 |
| fairness_threat | 公平受损 | 玩家觉得「不公平 / 被针对 / 规则偏向某方」 |
| trust_communication_gap | 信任与沟通落差 | 玩家觉得「官方不听、不说清、说到没做到」 |
| belonging_drop | 归属感下降 | 玩家觉得「这游戏 / 社区不再属于我」，疏离、想退坑 |
| norm_safety_risk | 规范与安全风险 | 涉及举报、封号、外挂、辱骂、骚扰等秩序与安全问题 |
| uncertain | 不确定 | 证据不足以判定机制 |

---

## competence_frustration（能力挫败）

- **构念定义**：玩家的胜任需求受挫——付出努力却达不到目标、被难度或数值劝退、进度停滞。
- **纳入标准**：出现「打不过 / 太肝 / 做不完 / 白练 / 投入无回报 / 门槛太高 / 卡关」等挫败表达。
- **排除标准**：挫败来源是「规则不公」而非「自身能力/投入」时归 `fairness_threat`；单纯抱怨「没时间」而无挫败情绪归 `uncertain`。
- **正例**：「八成打完一整把都不一定能做完任务」「肝了一周还是差一大截」。
- **反例**：「匹配到的人都比我强，一看就是代练」（→ fairness_threat）。
- **与相邻标签区别**：与 fairness_threat 的区别在于挫败**归因于自身/难度**而非**外部不公**。
- **多标签冲突**：若同时有难度挫败与公平质疑，取**证据文本主语气**；主语气不清则 `uncertain`。
- **是否允许无机制证据**：允许——纯「太难了」无进一步指向也可保留为弱证据，但短于最小长度或纯语气词判 no。

## fairness_threat（公平受损）

- **构念定义**：玩家感到规则、机制或对局结果对自己不公，或系统偏向特定群体（付费方、代练、某阵营）。
- **纳入标准**：「不公平 / 被针对 / 氪金才行 / 匹配不公 / 偏袒 / 暗改 / 坑」等。
- **排除标准**：仅表达自身打不过而无「不公」归因归 `competence_frustration`。
- **正例**：「让 3000 经济给对面，这不是明摆着坑我方吗」「氪佬随便虐，平民毫无机会」。
- **反例**：「我技术太差了」（→ competence_frustration）。
- **与相邻标签区别**：核心是「归因于外部规则/系统不公」。
- **多标签冲突**：与 competence 冲突时按主归因取一个；与 trust 冲突（既不公又质疑官方沟通）时，若重点在「结果不公」取 fairness，若重点在「官方不回应」取 trust。
- **是否允许无机制证据**：允许弱证据。

## trust_communication_gap（信任与沟通落差）

- **构念定义**：玩家对官方的信任受损——反馈无回应、公告含糊、承诺未兑现、改动不透明。
- **纳入标准**：「官方不听 / 没公告 / 暗改 / 说好的没做 / 客服无用 / 已读不回」等。
- **排除标准**：单纯抱怨机制本身而不涉及「官方沟通/信任」归对应机制。
- **正例**：「更新说明只字未提，玩家发现了才承认」「反馈了半年官方装死」。
- **反例**：「这英雄太强了」（→ 视表层，可能 balance + competence/fairness）。
- **与相邻标签区别**：核心在「官方—玩家之间的沟通/信任」而非机制数值本身。
- **多标签冲突**：与 fairness 共现时按主语气取一。
- **是否允许无机制证据**：允许。

## belonging_drop（归属感下降）

- **构念定义**：玩家的社群归属与身份认同下降——疏离、想退坑、「不再是我的游戏」、老玩家被冷落。
- **纳入标准**：「退坑 / 弃游 / 不想玩了 / 环境变了 / 老玩家寒心 / 圈子乌烟瘴气」等带情感疏离的表达。
- **排除标准**：单次挫败而无退坑/疏离倾向归 `competence_frustration`。
- **正例**：「玩了五年，现在是真的想走了」「这个社区已经不是当初的样子」。
- **反例**：「这把打得好累」（→ competence_frustration）。
- **与相邻标签区别**：核心是**关系/身份层面的疏离**，而非单次体验挫败。
- **多标签冲突**：疏离是结果、挫败是原因时——若文本重点在「想走/失望于社区」取 belonging_drop。
- **是否允许无机制证据**：允许。

## norm_safety_risk（规范与安全风险）

- **构念定义**：涉及游戏秩序与玩家安全的问题——外挂、代练、辱骂、骚扰、误封、举报无效、账号安全。
- **纳入标准**：「外挂 / 挂哥 / 代练 / 举报没用 / 被封号 / 辱骂 / 骚扰 / 盗号」等。
- **排除标准**：单纯数值/难度问题归对应机制。
- **正例**：「满屏都是挂，举报了也不处理」「无缘无故被封三天」。
- **反例**：「这版本太肝了」（→ competence_frustration）。
- **与相邻标签区别**：核心是**秩序/安全/合规**议题；虽常与 fairness 相关，但只要涉及外挂/封号/辱骂等秩序安全，优先归 norm_safety_risk。
- **多标签冲突**：与 fairness 共现（如「外挂横行不公平」）优先 norm_safety_risk（安全议题优先）。
- **是否允许无机制证据**：允许。

## uncertain（不确定）

- **构念定义**：证据文本不足以判定任一机制。
- **使用条件**：
  - 文本过短或纯语气（「没了」「没啥用」「哈哈」）；
  - 纯中性描述、无情绪/诉求指向；
  - 同时轻微指向多个机制且无法判定主机制；
  - 上下文缺失导致语义歧义。
- **要求**：`uncertain` 是合法且必要的编码，不得为覆盖率而回避使用。

---

## 冲突裁决优先级（当主语气不清时的兜底顺序）

1. 若涉及外挂/封号/辱骂/骚扰/账号安全 → `norm_safety_risk`；
2. 否则若核心是官方沟通/信任落差 → `trust_communication_gap`；
3. 否则若核心是外部规则/系统不公 → `fairness_threat`；
4. 否则若核心是退坑/疏离/身份 → `belonging_drop`；
5. 否则若核心是自身能力/投入挫败 → `competence_frustration`；
6. 仍不能判定 → `uncertain`。

> 该优先级仅为**主语气不清时**的兜底；文本主语气明确时以主语气为准。
